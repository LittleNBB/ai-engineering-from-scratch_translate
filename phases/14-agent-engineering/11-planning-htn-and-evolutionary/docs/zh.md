# HTN 规划与进化搜索

> 符号规划处理计划可证明正确的情况。进化代码搜索处理适应度函数可机器检查的情况。ChatHTN（2025）和 AlphaEvolve（2025）展示了与 LLM 配对时各自解锁的能力。

**类型：** Build
**语言：** Python（stdlib）
**前置课程：** Phase 14 · 02（ReWOO and Plan-and-Execute）
**时间：** ~75 分钟

## 学习目标

- 解释分层任务网络（Hierarchical Task Networks）：任务、方法、算子、前置条件、效果。
- 描述 ChatHTN 的混合循环 —— 带 LLM 兜底分解的符号搜索。
- 解释 AlphaEvolve 的进化循环以及为什么它只在有编程评估器时才有效。
- 用标准库实现一个玩具 HTN 规划器和一个玩具进化搜索。

## 问题

ReWOO（第 2 课）、Plan-and-Execute 和 ReAct 覆盖了大多数 Agent 规划。有两种情况它们覆盖得不好：

1. **具有可证明正确性的计划。** 调度、航线规划、合规工作流 —— 计划必须在构造上是合理的。一个流畅但偶尔幻觉出一个步骤的 LLM 计划是不可接受的。
2. **具有机器可检查适应度函数的优化。** 矩阵乘法、调度启发式、编译器遍历 —— 目标不是"一个正确的计划"而是"最佳计划"。

HTN 规划和 AlphaEvolve 解决了两个不同的问题。两者都将 LLM 用作放大器，而非替代品。

## 核心概念

### 分层任务网络（HTN）

一个 HTN 是：

- **任务** — 复合任务（待分解）和原始任务（直接可执行）。
- **方法** — 将复合任务分解为子任务的方式，带前置条件。
- **算子** — 带前置条件和效果的原始动作。
- **状态** — 一组事实。

规划：给定一个目标任务和初始状态，找到一个分解为原始算子的序列，使其前置条件按顺序满足。

HTN 比 LLM 更古老，仍然是可证明正确计划的参考。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交替进行：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，询问 LLM："在状态 `s` 下你如何分解 `task`？"
3. 将 LLM 响应转换为候选子任务。
4. 根据算子 schema 验证；拒绝无效分解。
5. 递归。

论文的核心声明：产生的每个计划都是可证明合理的，因为 LLM 建议仅作为候选分解进入，从不作为直接计划编辑。符号层负责正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续）添加了一个学习器，通过回归泛化 LLM 产生的分解 —— 将 LLM 查询频率降低高达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是另一个物种：由 Gemini 2.0 Flash/Pro 集成编排的进化代码搜索。

循环：

1. 从种子程序 + 编程评估器（返回适应度分数）开始。
2. LLM 集成提出变异。
3. 将变异通过评估器运行。
4. 保留最佳的；再次变异。

已发表的成果：

- 56 年来首次超越 Strassen 的 4x4 复数矩阵乘法改进（48 次标量乘法）。
- 通过 Borg 调度启发式恢复了 0.7% 的 Google 计算资源。
- 在前沿工作负载上 FlashAttention 加速 32%。

硬约束：适应度函数必须是机器可检查的。对散文答案的进化搜索不会收敛。

### 何时使用哪种

| 问题类别 | 使用 | 原因 |
|---------|------|------|
| 带硬约束的调度 | HTN + ChatHTN | 可证明的合理性 |
| 编译器优化 | AlphaEvolve | 机器可检查的适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无形式保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略绑定的自动化 | HTN | 前置条件编码策略 |

### 这个模式出错的地方

- **没有算子的 HTN。** 没有前置条件/效果 schema，合理性声明就崩塌了。ChatHTN 的"LLM 建议分解"需要 schema 来拒绝无效动作。
- **没有真实评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性的和快速的。
- **过度工程化。** 大多数 Agent 任务不需要两者。先尝试 ReAct 或 ReWOO。

## 构建它

`code/main.py` 实现了两个玩具：

- 一个带算子、方法、前置条件、效果的标准库 HTN 规划器，以及一个在没有方法匹配复合任务时启动的 `LLMFallback`。"LLM"是一个脚本化的分解器，使规划器可离线运行。
- 一个在算术程序上的标准库进化搜索：增长表达式，使其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行它：

```
python3 code/main.py
```

轨迹展示了 HTN 规划器分解复合任务（带中途 LLM 兜底）以及进化循环收敛到目标表达式。

## 使用它

- **HTN 规划器** — `pyhop`、`SHOP3`，或为领域特定策略构建自己的。
- **ChatHTN** — 研究代码；该模式（符号 + LLM 兜底）可以干净地移植到任何 HTN 规划器。
- **AlphaEvolve** — DeepMind 论文；该模式（集成 + 评估器）可复现。OpenEvolve 和类似的开源分支正在出现。
- **Agent 框架** — 目前还没有一等的 HTN 或 AlphaEvolve。作为子 Agent 或后台工作者构建。

## 发布它

`outputs/skill-hybrid-planner.md` 生成一个混合规划器脚手架（HTN 或进化），LLM 角色被显式限定。

## 练习

1. 用回溯扩展 HTN 规划器：当算子的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 给 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时先重新检查方法库。
3. 将进化搜索评估器换成真实的测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛所需代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合：用 HTN 将复合任务分解为子任务，然后对每个子任务的原始算子使用进化搜索。它在哪里发光，在哪里过度工程化？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| HTN | "分层规划器" | 带算子、前置条件、效果的任务分解 |
| Method | "分解规则" | 将复合任务拆分为子任务的方式 |
| Operator | "原始动作" | 带前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 当没有方法匹配时符号规划器询问 LLM |
| AlphaEvolve | "进化代码搜索" | LLM 集成变异代码；确定性评估器选择 |
| Fitness function | "评估器" | 对输出的确定性、机器可检查评分 |
| Online method learning | "缓存的 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
