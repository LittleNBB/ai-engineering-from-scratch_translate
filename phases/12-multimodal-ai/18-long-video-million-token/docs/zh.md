# 百万 Token 上下文下的长视频理解

> 一段 1 小时的 4K 视频以 24 FPS 经过 patch 和嵌入后，产生约 6000 万 token。2 小时播客节目转录后是 30,000 token。一部完整的蓝光电影，即使经过激进池化压缩，也是数十万 token。Google 的 Gemini 1.5（2024 年 3 月）以 1000 万 token 上下文开启了这个时代，能在小时级视频上做可靠的"大海捞针"检索。LWM（Liu 等人，2024 年 2 月）展示了 ring attention 的扩展路径。LongVILA 和 Video-XL 进一步扩展了摄取能力。VideoAgent 用代理式检索替代了原始上下文。每种方法在计算、召回和工程复杂度上都是不同的权衡。本课将它们并排解读。

**类型：** Build
**语言：** Python（标准库，大海捞针模拟器 + 代理式检索路由器）
**前置课程：** Phase 12 · 17（视频时间 token）
**时间：** ~180 分钟

## 学习目标

- 计算不同 FPS 和池化下长视频的总视觉 token 数。
- 解释三种扩展路径：暴力上下文（Gemini 1.5）、ring attention（LWM）、token 压缩（LongVILA / Video-XL）。
- 对比原始上下文视频 VLM vs 代理式检索视频 VLM（VideoAgent）在准确率和延迟上的差异。
- 为 30 分钟视频设计大海捞针测试并测量特定分钟的召回率。

## 问题

Qwen2.5-VL 大小的 patch 在 384 原生分辨率下每帧约 729 个 token。3×3 池化后每帧 81 个 token。30 分钟片段以 1 FPS = 1800 帧 = 145,800 个 token。2025 年开源 VLM 可行，但很紧。2 FPS 下 291,600 个 token——只有最大的上下文能容纳。

2 小时电影以 1 FPS 是 583k token。超出 2026 年大多数开源模型；需要 Gemini 2.5 Pro 或更激进的池化。

三条扩展路径应运而生。

## 概念

### 路径 1：暴力上下文（Gemini 1.5、Claude Opus）

用硬件堆砌解决问题。将上下文扩展到数百万 token，在一次前向传播中处理所有内容。

Gemini 1.5 Pro 以 100 万 token 上下文发布；Gemini 1.5 Ultra 扩展到 1000 万；2026 年的 Gemini 2.5 Pro 能可靠处理数小时视频。论文（arXiv:2403.05530）记录了在约 950 万 token 下 99.7% 的大海捞针召回率。

工程：使用内存层次结构（局部 + 全局 + 稀疏）的自定义注意力实现，加上 MoE 专家路由提升长上下文效率。未完整公开细节。未开源。

### 路径 2：Ring Attention（LWM、LongVILA）

Ring attention 将长序列分布到设备上形成"环"，每个设备持有一个块。全序列注意力通过每个设备以环形模式将其块发送给下一个设备来实现，计算部分注意力，然后聚合。

LWM（Liu 等人，2024）用这种方式训练了 100 万 token 上下文模型。训练计算与上下文线性增长，而非二次方——注意力的二次方开销被分摊到环上的设备。

LongVILA（arXiv:2408.10188）将该模式适配到 VLM。1400 帧视频每帧 192 个 token = 268k 上下文，使用 8 路并行 ring attention 训练。

### 路径 3：Token 压缩（Video-XL、LongVA）

比暴力上下文更便宜：在 LLM 看到序列之前激进压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要 token：每片段 N 帧产生一个"摘要" token，该 token 关注这 N 帧。推理时 LLM 每片段只看到一个摘要 token，大幅缩小上下文。

LongVA 通过"长上下文迁移"技术将 LLM 上下文从 200k 扩展到 200 万。在长上下文文本上训练，通过共享表示迁移到长上下文视频。

Token 压缩用特定时间戳的召回率换取可扩展性。模型大致知道发生了什么，但有时会错过确切帧。

### 路径 4：代理式检索（VideoAgent）

不将完整视频输入 LLM。相反，将视频视为数据库，用 LLM 查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 读取问题。
2. LLM 向检索工具请求相关片段（"给我看有猫的片段"）。
3. 工具返回匹配的片段时间戳。
4. LLM 通过 VLM 读取这些片段。
5. LLM 组织答案或提出后续查询。

这是 LLM-as-agent 模式应用于长视频。推理更便宜（只编码相关片段），工程更难（检索质量成为瓶颈）。

### 大海捞针基准

标准的长上下文测试：在视频的随机点插入一个独特的视觉或文本标记，然后提出需要回忆它的查询。

指标：跨视频长度和标记位置的 Recall@k。

Gemini 2.5 Pro 在长达 90 分钟的视频上得分 >99% 召回率。开源 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟上约 85-90%，超过 60 分钟退化。

VideoAgent 可以在 2 小时以上匹配或击败原始上下文模型，因为如果工具好，检索能命中针。

### 选择哪条路径

15 分钟片段追求前沿准确率：开源 72B + 原生上下文通常可行。选择 Qwen2.5-VL-72B。

30 分钟到 1 小时内容：开源选 LongVILA 或 Video-XL；闭源选 Gemini 2.5 Pro。质量标准很重要——前沿选闭源。

2 小时以上内容：VideoAgent 或类似检索模式。或者，将内容分层摘要然后提供层级式摘要。

### 2026 年生产模式

实际中，生产长视频流水线是混合式的：

1. 对整个视频运行动态 FPS 采样 + 激进池化（获得 100k token 的全局表示）。
2. 传给 72B VLM 生成全局摘要。
3. 如果用户问细节问题，使用摘要作为索引运行代理式检索。

这结合了暴力上下文用于全局理解和检索用于局部细节。

1. 45 分钟讲座，1 FPS，每帧 81 个 token。总 token 数？适配哪些模型的上下文？

2. 设计大海捞针测试：在第几分钟注入标记，确切的查询格式是什么？

3. 对比暴力上下文 Qwen2.5-VL-72B（80k 上下文）vs VideoAgent（Claude 3.5 + 检索）在 1 小时视频上。哪个在召回率上胜出？哪个在延迟上胜出？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Brute context（暴力上下文） | "就是更多 token" | 将 LLM 上下文扩展到数百万 token；一次处理所有内容 |
| Ring attention | "LWM 风格并行" | 每个设备持有一个块并旋转的分布式注意力模式 |
| Token compression（token 压缩） | "摘要 token" | 通过 LLM 之前的学习压缩器减少每片段 token 数 |
| Needle-in-haystack（大海捞针） | "NIH 测试" | 在随机点插入独特标记，要求模型在测试时回忆它 |
| Agentic retrieval（代理式检索） | "LLM 作为查询规划器" | LLM 向检索工具请求相关片段，通过 VLM 读取，组织答案 |
| VideoAgent | "视频检索模式" | 经典代理式检索设计：问题 → 工具 → 片段 → 答案 |
