# LLaVA-OneVision：一个模型覆盖单图像、多图像和视频

> 在 LLaVA-OneVision（Li 等人，2024 年 8 月）之前，开源 VLM 世界有不同的谱系：LLaVA-1.5 用于单图像，Mantis 和 VILA 等多图像模型，Video-LLaVA 和 Video-LLaMA 等视频模型。每个都在自己的基准上胜出，在其他基准上失败。LLaVA-OneVision 论证了单一课程可以训练一个模型在所有三种场景中占主导，而且涌现的任务迁移效果（单图像技能导出到视频，多图像推理导出到单图像）超过了专家模型的总和。方案看似简单：一个在场景间保持恒定的视觉 token 预算，加上一个从单图像到 OneVision（多图像）再到视频的显式课程。本课解读预算、课程和涌现行为。

**类型：** Build
**语言：** Python（标准库，token 预算求解器 + 课程规划器）
**前置课程：** Phase 12 · 05（LLaVA）、Phase 12 · 06（任意分辨率）
**时间：** ~180 分钟

## 学习目标

- 设计一个在单图像、多图像和视频输入间保持恒定的视觉 token 预算。
- 安排一个训练课程，将技能从单图像迁移到视频而不发生灾难性遗忘。
- 解释为什么在课程正确的前提下，单一模型在相同参数量下胜过专家模型。
- 说出 LLaVA-OneVision 报告的三种涌现能力：多摄像头推理、集合标记提示、iPhone 截图代理。

## 问题

图像、多图像和视频分别以不同方式考验模型。

单图像需要高分辨率 token（AnyRes，约 2880 个视觉 token）以捕捉 OCR 和细粒度细节。每样本预算：一张图像，2880 个 token。

多图像需要中等分辨率的多张图像（每张约 576 个 token），以便跨图像推理适配上下文。每样本预算：4-8 张图像，各 576 个，2300-4600 个 token。

视频需要低分辨率的多帧（池化后每帧约 196 个 token）以捕捉时间动态。每样本预算：8-32 帧，各 196 个，1600-6200 个 token。

如果训练独立模型，你选择一个预算。如果训练一个模型，你需要预算在场景间合理缩放而不撑爆上下文。

OneVision 之前的默认答案是"训练一个场景，忽略其他"。Video-LLaVA 通过额外训练阶段将视频嫁接到图像模型上。LLaVA-NeXT 通过拼贴添加了多图像支持。没有一个能干净地处理全部三种。

## 概念

### OneVision token 预算

LLaVA-OneVision 选择每样本约 3000-4000 个 token 的统一视觉 token 预算，按场景不同分配：

- **单图像**：AnyRes-9（3×3 瓦片 + 缩略图），每个瓦片 384 分辨率有 729 个 patch，激进双线性池化 2×2 → 每瓦片 182 个。总计：9 × 182 + 182 = 1820 个 token。或 AnyRes-4 每瓦片 729 = 2916 + 729。
- **多图像**：每张图像中等分辨率（384，无拼贴），729 个 token 无池化。预算 6 张图像 → 4374 个 token。
- **视频**：32 帧 384 分辨率配合激进 3×3 双线性池化 → 每帧 81 个 token。总计：32 × 81 = 2592 个 token。

分配维持大致恒定的总 token 数。LLM 永远不会看到撑爆上下文的 batch。编码器为每种场景产生不同的几何形状，但 LLM 消费相同的预算。

### 三阶段课程

LLaVA-OneVision 分三阶段训练：

1. **单图像 SFT（阶段 SI）**。所有数据都是单图像加文本。在高分辨率 AnyRes 输入上训练。这教会感知、OCR 和细粒度理解。使用 LLaVA-NeXT 数据加上 OneVision 特定的单图像数据。
2. **OneVision SFT（阶段 OV）**。混合单图像 + 多图像 + 视频（均匀采样帧）。在统一 token 预算上训练。这教会模型处理异构 batch 形状。不重置权重——从阶段 SI 继续。
3. **任务迁移（阶段 TT）**。继续使用目标任务组合，通常根据产品侧重多图像或视频。可选的部署微调。

关键：课程顺序很重要。先训练视频或多图像会比先训练单图像产生更差的图像性能，即使数据相同。论文明确消融了这一点。

### 为什么课程有效

单图像训练构建感知基础。patch token 携带细粒度视觉特征；LLM 学习将它们与文本整合。多图像和视频引入结构性挑战（哪张图是哪个，什么先发生），没有强感知基础很难学会。

如果从头一起训练所有场景，模型在感知上欠拟合（每 batch 单图像数据有限）在结构上过拟合（大量多图像/视频数据）。结果：一个遵循跨图像推理模式但视觉浅薄的模型。

课程排序给你阶段 SI 的感知强度，然后阶段 OV 的组合/时间推理，两者都不丢失。

### 涌现的跨场景技能

LLaVA-OneVision 论文报告了三种涌现能力：

1. **多摄像头推理**。分别在多图像 + 视频上训练；推理时被要求推理多摄像头驾驶场景。模型正确整合了视角，尽管训练中从未见过这种确切格式。
2. **集合标记提示**。用户用编号标记注释图像中的物体；模型推理"标记 3 相对于标记 7 在做什么"。训练中既没有标记也没有注释；从空间定位 + 多图像引用的组合中学到。
3. **iPhone 截图代理**。用户提供 iPhone 屏幕截图并要求规划下一次点击。在 UI 截图、用户工作流视频和多图像前后配对上训练。泛化到代理用例。

这些不是训练任务；它们从课程的组合结构中涌现。

### 视觉 token 池化

token 预算需要池化。OneVision 在二维 patch 网格上使用双线性插值：24×24 = 576 个 patch 变成 12×12 = 144（2 倍因子）或 8×8 = 64（3 倍因子）。池化在 patch 网格空间而非 token 空间中进行，以保持局部性。

每场景的池化因子选择本身是一个超参数。更少池化 = 更多 token = 更丰富表示。更多池化 = 更少 token = 更多帧/图像适配。

### LLaVA-OneVision-1.5

2025 年的后续（LLaVA-OneVision-1.5，arXiv 2509.23661）在训练数据、模型权重和代码上是"完全开放的"。在部分基准上匹配了专有差距并民主化了该方案。相同课程，更多数据，更好的基础 LLM。没有架构变化。

### 与 Qwen2.5-VL 的对比

Qwen2.5-VL（课程 12.09）做了不同的选择。它使用 M-RoPE 和动态 FPS 而非固定池化。其预算随输入缩放——1 分钟的视频比 5 秒视频使用更多 token。LLaVA-OneVision 固定预算并缩放池化。两者都有效；它们在可配置性和可预测性之间做了权衡。

## 使用它

`code/main.py` 是 OneVision 风格 VLM 的课程和预算规划器。给定每样本 token 预算和目标场景组合（比如 40% 单图像、30% 多图像、30% 视频），它：

- 为每场景分配分辨率、池化因子和帧数。
- 检查每场景是否在共享预算内。
- 报告预期 token 数、LLM FLOPs 和哪些场景 token 不足。
- 打印逐阶段训练调度。

用它规划 OneVision 微调或检查 VLM 部署的每请求成本。

## 练习

1. 你的产品支持 80% 单图像、10% 多图像（2-4 张图）、10% 视频（8-16 帧）。设计 token 预算。你会把不做重度多图像节省下来的额外预算放在哪里？

2. 阅读 LLaVA-OneVision 第 4.3 节（涌现能力）。提出课程可能解锁但论文未报告的第四种涌现技能。

3. 交换课程顺序——先训练多图像，然后单图像，然后视频。预测哪些基准会退化及原因。

4. 论文报告视频基准仅在每样本 8 帧上训练。这能泛化到推理时的 30 秒视频吗？什么先崩溃——token 预算还是时间推理？

5. 将 24×24 patch 双线性池化到 12×12 是每维 4 倍的缩减。用标准库 Python 实现池化并验证每个 2×2 块的均值与双线性输出匹配。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| OneVision scenario（OneVision 场景） | "单图像、多图像或视频" | 统一 VLM 处理的三种输入形式之一；预算在三者间保持恒定 |
| Token budget（token 预算） | "每样本多少 token" | LLM 每个训练/推理样本看到的总视觉 token，通常 3000-4000 |
| Curriculum（课程） | "训练顺序" | 为涌现迁移选择的阶段排序（单图像 → 多图像 → 视频） |
| Bilinear pooling（双线性池化） | "token 收缩" | 对 patch 网格（2D）应用双线性插值以减少 token 数同时保持局部性 |
| Emergent skill（涌现技能） | "没训练过，仍然有效" | 在推理时出现但没有匹配训练数据的能力，源于课程组合 |
| AnyRes-k | "k 瓦片设置" | k 个固定分辨率的子瓦片加一个缩略图，典型 k ∈ {4, 9} |
| Task transfer（任务迁移） | "跨场景泛化" | 在单图像上学到的技能通过共享骨干网络应用到视频（反之亦然） |
